# FunctionGemma 置信度计算方案

> 前提认知：**FunctionGemma 本身不输出置信度数字**。它只是个语言模型，吐 token 序列。
> 「置信度」是在模型外部由调用方计算出来的，用于决定一条指令走端侧还是兜底到云端 AutoGLM。
> 配套架构见 [hybrid-agent-routing.md](./hybrid-agent-routing.md)。

## 三层方案（由硬到软）

### 第 1 层 · 结构性硬门控（主门控，必做）

不算概率，只校验输出是否合法。离线小模型最可靠的信号，且**仅依赖已有的 JSON Schema 校验**，LiteRT 上即可实现。

判定（任意一条不过 → 低置信 → 转 AutoGLM）：

1. 是否产出 `<start_function_call>`？
   **未产出（退化为纯文本回复）= 模型自认不会 → 这是最天然的「我不确定」信号。**
2. 函数名是否在工具白名单内？
3. `required` 参数是否齐全、类型与 schema 匹配？

### 第 2 层 · Token 概率 / logprob（软置信，可选）

运行时若能拿到 logprobs，计算生成该 tool_call 的概率：

- 取**函数名 token** 概率，或整串 arguments 的**几何平均概率**（长度归一化，避免长输出被惩罚）。
- 阈值示例：几何平均概率 < 0.6 判低置信。
- ⚠️ 限制：LiteRT-LM 对外暴露 logprobs 的接口有限，端侧不一定取得到。取不到则退回第 1 层。

### 第 3 层 · 自一致性采样投票（可选，对误触发敏感的动作）

同一指令带温度采样跑 N 次（N=3~5）：

- N 次同函数名 + 同关键参数 → 高置信。
- 函数名分歧 → 低置信 → 转云端。
- 代价 N 倍延迟；适合 `set_screen_power` 等误触发代价高的动作。

## 决策流

```
FunctionGemma 输出
   │
   ├─ 第1层 硬门控：合法 tool_call 且命中白名单且参数齐全？
   │     否 ───────────────────────────► AutoGLM 云端
   │     是
   │     ▼
   ├─ 第2层 软置信（能取 logprob 才做）：
   │     函数名/参数几何平均概率 ≥ 阈值？
   │     否 ───────────────────────────► AutoGLM 云端
   │     是
   │     ▼
   └─ 端侧直接执行
```

## 落地建议

- **优先把第 1 层做扎实**：不依赖任何概率接口，纯 JSON Schema 校验即可，已能挡掉绝大多数「该上云却端侧硬答」。
- 第 2、3 层是锦上添花，且受 LiteRT-LM 运行时接口限制。
- 待确认：当前 LiteRT-LM 版本能否取 logprobs，决定第 2 层是否可落地。

## 诚实边界

FunctionGemma 官方 model card 未承诺暴露 token 概率，第 2 层能否实现取决于所用 LiteRT-LM 运行时版本。
